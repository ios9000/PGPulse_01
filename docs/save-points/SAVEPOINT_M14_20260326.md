### What PGPulse Does

PGPulse is a real-time PostgreSQL monitoring, alerting, anomaly detection, root cause analysis, and guided remediation tool built in Go. It connects to one or more PostgreSQL instances via pg_monitor role, collects ~148 metrics (ported from 76 legacy PGAM queries plus new collectors), stores time-series data in PostgreSQL/TimescaleDB, evaluates 23 alert rules (including ML-based forecast alerts), performs automated root cause analysis with 20 causal chains, generates remediation recommendations via 25 adviser rules, and provides step-by-step guided remediation playbooks — all through an embedded React web UI with 75 API endpoints.

PGPulse implements a four-layer operational stack: (1) Alerting — threshold and forecast-based alerts with email notifications; (2) Adviser — rule-based remediation recommendations triggered by metrics and alerts; (3) Root Cause Analysis — causal graph traversal with 20 chains, ML and threshold-based anomaly detection, incident timelines, confidence scoring, and auto-trigger on CRITICAL alerts; (4) Guided Remediation Playbooks — step-by-step executable playbooks with inline SQL execution, tier-based safety (diagnostic/remediate/dangerous/external), branching logic, concurrency guards, and implicit feedback collection. DBAs can cancel/terminate sessions, run on-demand EXPLAIN plans, compare pg_settings across instances, browse auto-captured query plan history, view temporal settings timelines, monitor logical replication, and follow guided playbooks to resolve incidents.

It supports PostgreSQL 14-18 via version-adaptive SQL gates, runs as a single binary with the frontend embedded via go:embed, provides JWT authentication with 4-role RBAC, and optionally runs as a desktop application via Wails v3 with system tray integration. Instances can be managed through the web UI (add/edit/delete, CSV bulk import) with YAML seeding on startup and orchestrator hot-reload every 60 seconds.

### Origin Story

Rewrite of PGAM — a legacy PHP PostgreSQL Activity Monitor used internally at VTB Bank.
PGAM had 76 SQL queries across 2 PHP files (analiz2.php + analiz_db.php), zero auth,
SQL injection vulnerabilities, and relied on COPY TO PROGRAM for OS metrics (superuser required).
PGPulse is a clean-room rewrite in Go that preserves the SQL monitoring knowledge while
fixing every architectural and security flaw.

### Key Design Decisions

| # | Decision | Rationale | Date |
|---|----------|-----------|------|
| D1 | Single binary deployment | Simplicity, go:embed frontend | 2026-02-25 |
| D2 | pgx v5 (not database/sql) | Named args, COPY protocol, pgx-specific features | 2026-02-25 |
| D3 | Version-adaptive SQL via Gate pattern | Support PG 14-18 without code branches | 2026-02-25 |
| D4 | No COPY TO PROGRAM ever | PGAM's worst security flaw — eliminated entirely | 2026-02-25 |
| D5 | pg_monitor role only | Least privilege, never superuser | 2026-02-25 |
| D6 | One Collector per module | Testable, enable/disable, independent intervals | 2026-02-25 |
| D7 | React 18 + TypeScript | Type safety, ecosystem maturity | 2026-03-03 |
| D8 | pgxpool.Pool per instance | Eliminates connection contention between collectors | 2026-03-04 |
| D9 | DB is source of truth for instances | YAML seeds on first start, DB overrides after | 2026-03-04 |
| D10 | DBCollector parallel to Collector | Per-database metrics without touching instance-level interface | 2026-03-08 |
| D11 | Simplified STL (EWMA + folded mean) | Full Loess too complex for pure-Go implementation | 2026-03-09 |
| D12 | mlerrors package breaks circular import | alert cannot import ml; shared sentinel errors in mlerrors | 2026-03-09 |
| D13 | Sustained crossing for forecast alerts | N consecutive points must cross threshold before alert fires | 2026-03-09 |
| D400 | RCA output: Incident Timeline + Lightweight Causal Graph | Graph is internal reasoning model; timeline is user-facing explanation | 2026-03-21 |
| D401 | RCA trigger: Auto on CRITICAL + on-demand for any | CRITICAL alerts get automatic RCA; user can trigger for WARNING or manual investigation | 2026-03-21 |
| D402 | RCA time window: 30-min lookback (configurable to 2h) | Most PG incident chains fit in 30 min | 2026-03-21 |
| D403 | UI output: inline summary on alert detail + dedicated RCA page | Inline is cheap, full page for deep investigation | 2026-03-21 |
| D404 | ML dependency: ML-enhanced, not required | With ML: use Z-scores. Without: rate-of-change threshold fallback | 2026-03-21 |
| D405 | 3 sub-iterations: Engine Reliability, UI, Expansion/Calibration | Staged delivery with increasing capability | 2026-03-21 |
| D406 | 20 causal chains, tiered A/B | Core PG failure modes; Tier A in M14_01, Tier B activated in M14_03 | 2026-03-21 |
| D407 | Negative evidence required in M14_01 | RCA must prune unsupported branches; otherwise false positives unacceptable | 2026-03-21 |
| D408 | Shared RCA + Adviser ontology | RCA chains and adviser rules evolve from shared knowledge base with stable identifiers and remediation hooks | 2026-03-21 |
| D500 | M14_03 tier ordering (W1-W10 as specified) | Calibration before expansion, integration last | 2026-03-22 |
| D501 | Comprehensive ML metric mapping | All metrics any chain references must be tracked, not just minimum-to-fire | 2026-03-22 |
| D502 | 2 agents for M14_03 (Backend + Frontend) | No file overlap, parallel execution | 2026-03-22 |
| D503 | Threshold fallback: 4h window + 15min calm period | Robust threshold calibration for RCA without ML | 2026-03-22 |
| D504 | JSON tags on CausalNode/CausalEdge with seconds conversion | Frontend needs duration fields in seconds for display | 2026-03-22 |
| D505 | RCA-Adviser bridge uses Unified Upsert | Not a separate bridge type; extends existing Upsert | 2026-03-22 |
| D506 | Schema uses BIGINT[] + urgency_score FLOAT8 | Not JSONB + boolean; migration 017 | 2026-03-22 |
| D507 | EvaluateHook() lives on remediation.Engine | Not a new type; engine extended | 2026-03-22 |
| D508 | Migration 017 includes partial unique index + GIN index | Performance for RCA-linked queries | 2026-03-22 |
| D600 | M14_04 under M14 (RCA family) | Playbooks are the action layer for RCA findings | 2026-03-24 |
| D601 | Database-stored playbooks, editable from UI | Maximum flexibility; seed pack for bootstrapping | 2026-03-24 |
| D602 | Four-tier execution safety model | Diagnostic (read-only), Remediate (DML), Dangerous (DDL/kill), External (manual) | 2026-03-24 |
| D603 | Playbook steps with SQL templates, safety tiers, result interpretation, branching | Declarative step definitions; no scripting engine | 2026-03-24 |
| D604 | Playbook Resolver with 5-level priority ranking | hook > root_cause > metric > adviser_rule > manual | 2026-03-24 |
| D605 | Core 10 seed pack | WAL, replication, connections, locks, long-tx, checkpoint, disk, vacuum, wraparound, query | 2026-03-24 |
| D606 | Bounded conditional branching, no loops/nesting | Linear step progression with optional skip-ahead; prevents infinite loops | 2026-03-24 |
| D607 | Static declarative rules for result interpretation | Column comparisons and row-count rules; no dynamic scripting | 2026-03-24 |
| D608 | 2 agents (Backend + Frontend) for M14_04 | Same parallel strategy as M14_03 | 2026-03-24 |
| D609 | PlaybookRun persisted with resume; approval queue deferred | Core run persistence; notification/delegation deferred | 2026-03-24 |

## 7. KEY LEARNINGS & DECISIONS LOG

### Architecture Decisions

| Date | Decision | Alternatives Considered | Why This Choice |
|---|---|---|---|
| 2026-02-25 | Go over Rust | Rust has steeper learning curve | Faster development, goroutines natural for collectors |
| 2026-02-25 | pgx over database/sql | database/sql lacks PG-specific features | Named args, COPY, notifications |
| 2026-02-25 | TimescaleDB over InfluxDB | InfluxDB requires separate service | PG-native, one less dependency |
| 2026-02-25 | Agent Teams (4 agents) | 7 agents (enterprise template) | Right-sized for 1-dev project |
| 2026-03-01 | Agents run bash directly | Hybrid mode (files only) | EINVAL bug fixed in Claude Code v2.1.63 |
| 2026-03-03 | React 18 over Svelte | Svelte ecosystem smaller | React ecosystem maturity, TypeScript support |
| 2026-03-04 | pgxpool.Pool per instance | Shared pool | Eliminates connection contention |
| 2026-03-04 | DB source of truth for instances | YAML-only config | Enables UI-based instance management |
| 2026-03-08 | DBCollector parallel to Collector | Merged interface | Clean separation of instance vs per-database |
| 2026-03-09 | EWMA + folded mean STL | Full Loess | Pure Go complexity constraint |
| 2026-03-09 | mlerrors breaks circular import | Interface wrapping | Simplest solution for alert->ml dependency |
| 2026-03-09 | Sustained crossing for forecast alerts | Single-point threshold | Reduces false positives dramatically |
| 2026-03-13 | Rule-based remediation (25 rules) | ML-based recommendations | Deterministic, auditable, immediate value |
| 2026-03-16 | PGSS snapshots with diff engine | Live-only query stats | Enables workload comparison and regression detection |
| 2026-03-17 | Wails v3 with build tags | Electron | Go-native, single binary, no JS runtime |
| 2026-03-21 | RCA: Causal graph + incident timeline | Simple rule chains | Graph enables pruning, confidence scoring, and expansion |
| 2026-03-21 | RCA: ML-enhanced but not required | ML-only anomaly detection | Works without ML via threshold fallback |
| 2026-03-21 | 20 causal chains with Tier A/B | Start with fewer | Comprehensive coverage of PG failure modes |
| 2026-03-22 | Threshold fallback: 4h + 15min calm | Simpler thresholds | Avoids false positives from noisy baseline |
| 2026-03-22 | Unified Upsert for RCA-Adviser bridge | Separate bridge type | Reuses existing remediation infrastructure |
| 2026-03-24 | Playbooks: DB-stored, editable from UI | Config-file playbooks | Maximum flexibility, version control via DB |
| 2026-03-24 | Four-tier safety model | Binary safe/unsafe | Graduated permissions match DBA workflows |
| 2026-03-24 | Transaction-scoped SQL execution | Session-level SET | Prevents connection pool poisoning on panic/timeout |
| 2026-03-24 | Multi-statement injection guard | Trust SQL templates | Defense-in-depth against template manipulation |
| 2026-03-24 | Interpreter scope "first" only (MVP) | any/all evaluation | Single-row authoring rule eliminates need for complex logic |
