### What PGPulse Does

PGPulse is a PostgreSQL monitoring system written in Go that provides real-time visibility into PostgreSQL instance health, performance, and activity. It monitors connections, cache efficiency, replication lag, lock contention, query performance (via pg_stat_statements), vacuum/maintenance progress, and long-running transactions across PostgreSQL 14–17 instances.

The system is designed as a single-binary deployment with embedded web UI, time-series metric storage (TimescaleDB), configurable alerting, and future ML-based anomaly detection. It replaces a legacy PHP application (PGAM) that had zero authentication, SQL injection vulnerabilities, relied on superuser COPY TO PROGRAM for OS metrics, and used point-in-time rendering with no historical data.

### Origin Story

Rewrite of PGAM — a legacy PHP PostgreSQL Activity Monitor used internally at VTB Bank. PGAM had 76 SQL queries across 2 PHP files (analiz2.php + analiz_db.php), zero auth, SQL injection vulnerabilities, and relied on COPY TO PROGRAM for OS metrics (superuser required). PGPulse is a clean-room rewrite in Go that preserves the SQL monitoring knowledge while fixing every architectural and security flaw.

### Key Design Decisions

| # | Decision | Rationale | Date |
|---|----------|-----------|------|
| D1 | Single binary deployment | Simplicity, go:embed frontend | 2026-02-25 |
| D2 | pgx v5 (not database/sql) | Named args, COPY protocol, pgx-specific features | 2026-02-25 |
| D3 | Version-adaptive SQL via Gate pattern | Support PG 14–18 without code branches | 2026-02-25 |
| D4 | No COPY TO PROGRAM ever | PGAM's worst security flaw — eliminated entirely | 2026-02-25 |
| D5 | pg_monitor role only | Least privilege, never superuser | 2026-02-25 |
| D6 | One Collector per file (granular) | Testable, enable/disable, independent intervals | 2026-02-25 |
| D7 | Hybrid agent workflow | Claude Code bash broken on Windows; agents create files, dev runs bash | 2026-02-25 |
| D8 | InstanceContext with IsRecovery (SSoT) | Orchestrator queries pg_is_in_recovery() once per cycle, passes to all collectors | 2026-02-25 |
| D9 | Explicit collector registration in main.go | No init()/auto-registration magic; clear, debuggable | 2026-02-25 |
| D10 | Stateful collector pattern (checkpoint.go) | computeMetrics() pure function, sync.Mutex, detects stats reset via counter decrease | 2026-02-26 |
| D11 | pgssAvailable() loose coupling | Helper in base.go, not in InstanceContext; returns false → nil, nil | 2026-02-26 |
| D12 | Q50+Q51 combined single query | ROW_NUMBER + CROSS JOIN totals, "other" bucket in Go, fixed LIMIT 20 | 2026-02-26 |
| D13 | Lock tree: pg_blocking_pids() + Go BFS | Replaces PGAM's recursive SQL CTE; simpler, testable, cycle-safe | 2026-02-26 |
| D14 | Lock tree emits summary metrics only | Full tree structure deferred to M2/API; MetricPoint is numeric | 2026-02-26 |
| D15 | Q56+Q57 merged with parameterized threshold | Single query with CASE, $1 interval param, hardcoded "5 seconds" constant | 2026-02-26 |
| D16 | backend_type='client backend' filter | Wait events + long txns exclude background workers — improvement over PGAM | 2026-02-26 |
| D17 | Checkpoint PG 17 version gate | PG ≤ 16 → pg_stat_bgwriter. PG ≥ 17 → pg_stat_checkpointer CROSS JOIN pg_stat_bgwriter. -1 sentinel for unavailable columns | 2026-02-26 |
| D18 | pg_stat_io: stateless, COALESCE to -1 | PG < 16 returns nil, nil. Labels: backend_type, object, context. Skip metrics where value < 0 | 2026-02-26 |

### Key Interfaces (collector.go)

```go
type InstanceContext struct {
    IsRecovery bool
}

type MetricPoint struct {
    InstanceID string
    Metric     string
    Value      float64
    Labels     map[string]string
    Timestamp  time.Time
}

type Collector interface {
    Name() string
    Collect(ctx context.Context, conn *pgx.Conn, ic InstanceContext) ([]MetricPoint, error)
    Interval() time.Duration
}

type MetricStore interface {
    Write(ctx context.Context, points []MetricPoint) error
    Query(ctx context.Context, query MetricQuery) ([]MetricPoint, error)
    Close() error
}

type AlertEvaluator interface {
    Evaluate(ctx context.Context, metric string, value float64, labels map[string]string) error
}
```

## 7. KEY LEARNINGS & DECISIONS LOG

### Architecture Decisions

| Date | Decision | Alternatives Considered | Why This Choice |
|---|---|---|---|
| 2026-02-25 | Go over Rust | Rust steeper learning curve | Faster development, goroutines natural for collectors |
| 2026-02-25 | pgx over database/sql | database/sql lacks PG-specific features | Named args, COPY, notifications |
| 2026-02-25 | TimescaleDB over InfluxDB | InfluxDB requires separate service | PG-native, one less dependency |
| 2026-02-25 | Agent Teams (right-sized per task) | Fixed 4-agent team for everything | 2 agents for M1_05 worked well; scale to task complexity |
| 2026-02-25 | Hybrid mode (agents + manual bash) | Pure Agent Teams | Windows bash bug forced this |
| 2026-02-25 | Granular collectors (one per file) | Monolithic collector per category | Testable, independent intervals, enable/disable |
| 2026-02-26 | InstanceContext SSoT for IsRecovery | Each collector queries pg_is_in_recovery() | Single query per cycle, consistent state |
| 2026-02-26 | Explicit registration in main.go | init()/auto-registration | Debuggable, no magic |
| 2026-02-26 | Stateful collector pattern | Recompute rates from absolute values each time | Accurate rates, handles stats reset |
| 2026-02-26 | pg_blocking_pids() + Go BFS | Recursive SQL CTE (PGAM approach) | Testable, cycle-safe, simpler |
| 2026-02-26 | Summary metrics for locks, full tree deferred to API | Full tree in MetricPoint | MetricPoint is numeric; tree is structural |
